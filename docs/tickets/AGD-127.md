# AGD-127 — Implement Cache Invalidation on Appointment Actions

## Objective

Ensure cached appointment-related views refresh immediately after appointment create, update, and cancel actions so dashboard, queue, and reports do not show stale data.

## Affected Cache Scope

- Backend centralized cache namespaces:
  - `dashboard`
  - `reports`
  - `queue`
- Frontend short-term cache namespaces:
  - `appointment-admin-master-list`
  - `appointment-patient-list`
  - `appointment-medical-history` when status becomes `completed`
  - `appointment-recycle-bin` when an appointment is canceled or restored
  - `appointment-admin-by-date`
  - `appointment-admin-calendar`
  - `appointment-calendar-detail`
  - `appointment-patient-today-queue`
  - `appointment-admin-today-queue`
  - shared admin dashboard/report caches

## Fix Applied

- Deferred Laravel cache invalidation listeners until `afterCommit` so cache versions are bumped only after appointment transactions succeed.
- Kept appointment create, update, and cancel flows covered by the same dashboard/report/queue invalidation hooks.
- Split Flutter appointment mutation invalidation into targeted cache groups so appointment actions clear the relevant read models without evicting unrelated service catalog data.
- Added regression tests for:
  - create appointment cache refresh
  - update appointment cache refresh
  - cancel appointment cache refresh
  - frontend mutation cache invalidation coverage
